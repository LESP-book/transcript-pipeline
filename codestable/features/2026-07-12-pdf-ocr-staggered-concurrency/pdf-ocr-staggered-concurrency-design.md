# PDF OCR 连续错峰并发设计

## 0. 术语约定

- **连续错峰投递**：每次只新增一个单页请求，相邻请求启动时间至少间隔配置秒数，但不等待前一个请求完成。
- **页任务**：由页码、总页数和单页图片组成的一次独立 OCR 请求。
- **结果槽**：按 PDF 原始页码索引保存 OCR 文本的位置；请求完成顺序不会改变最终拼接顺序。

## 1. 决策与约束

### 需求摘要

当前实现必须等待上一页 OCR 完成才发送下一页，长 PDF 总耗时接近所有页面耗时之和。改为多页独立投递时，还要避开 codex-lb 的并发流限制，并保证乱序返回不会打乱正文。

成功标准：

1. 默认每 5 秒投递一个页任务，不等待前页完成，已投递页面可同时执行。
2. 客户端并发限制和投递间隔均可配置；并发值 `0` 表示不限制活动请求数，正整数用于显式限流。
3. 任意完成顺序下，最终文本严格按原始页码拼接，跨页不额外插入换行。
4. 任一页失败后停止投递尚未启动的页面，不写入不完整 sidecar，并明确报告失败页码。
5. 日志可看到页任务的投递、完成和失败进度。

明确不做：

- 不把多张图片重新打包到同一个请求。
- 不加入自动重试、静默跳页或部分成功输出。
- 不改变 OCR 文本清洗规则和下游中间产物结构。

投递间隔有明确依据：既有单请求多图会稳定触发上下文长度错误，用户已调整 codex-lb 并发限制，并明确要求每次只投递一张图片、每 5 秒继续投递且不等待前页完成。因此默认不设置客户端活动请求上限，只用 5 秒间隔抑制瞬时突发。

## 2. 名词与编排

### 2.1 名词层

**现状**：`run_codex_api_pdf_ocr` 在一个 `for` 循环内同步发送每一页；返回文本直接追加到列表，因此请求与完成均严格串行。

**变化**：新增页任务调度器。调度器只负责启动节流、活动任务计数、结果槽写入和失败传播；具体 Codex payload 构造及文本清洗仍由参考资料模块负责。

新增配置：

```yaml
reference:
  codex_ocr_max_concurrency: 0
  codex_ocr_submit_interval_seconds: 5.0
```

### 2.2 编排层

```mermaid
flowchart LR
  A[PDF 转换为带页码的单页图片] --> B[连续错峰调度器]
  B -->|每 5 秒| C[新增一个单页 OCR 请求]
  C --> D[结果按任意顺序返回]
  D --> E[按页码写入结果槽]
  E --> F{全部页面成功?}
  F -->|是| G[按页码无额外换行拼接]
  F -->|否| H[停止新投递并报告失败页]
  G --> I[写入完整 sidecar]
```

跨层纪律：

- 调度器不预先把全部页面排进线程池，只在间隔到达时新增一页；如果配置了正整数并发上限，还要等待活动槽位。
- 每个页任务继续携带“当前页/总页数”，服务端返回不依赖请求到达顺序。
- 已经启动的任务在某页失败后允许自然结束，但其结果不会形成部分 sidecar。
- 线程池仅用于网络 I/O；共享客户端不保存单次请求状态，每次调用内部独立构造请求与响应解析状态。

### 2.3 挂载点清单

- 配置模型与两份默认配置：新增并发数和投递间隔。
- OCR 页调度器：负责连续错峰、乱序回收及失败页标记。
- 参考资料 OCR：构造页任务、记录进度并保持原有文本拼接和 sidecar 语义。
- 测试与 README：覆盖默认配置、并发上限、错峰启动、乱序重排和失败不落盘。

### 2.4 推进策略

1. 配置契约：加入可验证的并发与间隔配置。退出信号：默认值可加载，非法负值被拒绝。
2. 调度编排：实现无预排队的连续错峰调度并接入 OCR。退出信号：乱序完成仍按页输出，失败页可追踪且不再投递新页。
3. 验证收尾：补齐针对性、全量与真实样例验证。退出信号：测试通过且不改变 sidecar 和跨页拼接契约。

## 3. 验收契约

1. 三页任务中第 2 页先完成、第 1 页后完成 → 最终输出仍为第 1、2、3 页顺序。
2. 配置并发数为 0、间隔为 5 秒 → 即使前页仍未完成，也在 0、5、10、15 秒继续投递后续页。
3. 配置正整数并发上限 → 活动请求达到该值后等待空槽，并继续保持最小启动间隔。
4. 第 N 页失败且后续页尚未投递 → 后续页不再启动，异常包含第 N 页，sidecar 不存在。
5. 相邻两页分别以连续句子的前后半句结束/开始 → 拼接处不人为加入换行。
6. 反向核对：没有加入自动重试、页数上限、文本截断或失败后静默跳页。

## 4. 与项目级架构文档的关系

当前仓库没有 `codestable/architecture/` 或 `codestable/requirements/` 基线文档。本功能只改变 PDF OCR 页任务的运行编排，不改变外部 API 或中间产物主结构。
