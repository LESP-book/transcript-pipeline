# 多页 PDF OCR 上下文超限根因分析

## 1. 问题定位

| 关键位置 | 说明 |
|---|---|
| `src/reference_utils.py:493` | 将 PDF 每一页渲染为 PNG data URL，但只返回 URL 列表；未校验渲染页号是否连续或是否与 PDF 原始页数一致。 |
| `src/reference_utils.py:579-600` | 将所有页面图片追加到同一个 `content` 列表，并只调用一次 `responses_stream_text()`。多页图像的完整内容因此进入单一模型上下文。 |
| `src/reference_utils.py:604-610` | 原实现仅校验合并后的整体文本非空；即使服务端未识别某一页，也无法定位或阻止不完整结果写入 sidecar。 |

## 2. 失败路径还原

**正常路径**：单页 PDF → 渲染一张 PNG → 构造一个包含一张图片的请求 → codex-lb 转发 → 返回该页 OCR 文本。

**失败路径**：多页扫描 PDF → 渲染多张高分辨率 PNG → 所有图片都被追加到同一请求 → codex-lb 转发给模型时输入超过上下文窗口 → 返回 `context_length_exceeded` → 没有 OCR 输出。

**分叉点**：`src/reference_utils.py:579-584` — 循环把每一页图像写入同一个请求，而非为每页创建一个独立请求。

## 3. 根因

**根因类型**：logic。

**根因描述**：实现将“PDF 的页列表”错误地等同于“一次模型输入”。扫描 PDF 的图片内容会随页数累积，单次请求的多模态上下文大小随之增长，最终超过模型窗口。实现也没有为每一页建立完成条件，因此无法证明最终文本覆盖了所有已渲染页面。

**是否有多个根因**：是。主因是多页图片被合并进一次请求；次因是渲染结果按文件名字符串排序且未验证连续页号、未逐页校验 OCR 返回值，十页以上 PDF 可能出现错误顺序或未被发现的缺页。

## 4. 影响面

- **影响范围**：所有通过 `codex_api` 处理的多页扫描 PDF，页数或图像大小足够大时均会触发；单页 PDF 不受主因影响。
- **潜在受害模块**：`run_codex_api_pdf_ocr()` 的直接调用、阶段 3 的参考原文提取、Web 任务的 PDF OCR。
- **数据完整性风险**：当前请求若返回部分文本但整体不为空，会被当作成功并写入 sidecar，无法知道是否漏页；页数超过 9 时，字符串排序还可能打乱页序。
- **严重程度复核**：维持 P1。多页扫描 PDF 是该功能的核心输入类型，且当前稳定失败或可能静默产生不完整结果。

## 5. 修复方案

### 方案 A：逐页串行 OCR，并在合并前校验全部页

- **做什么**：读取 PDF 预期页数；渲染后按数值页号排序并验证每一页图片均存在；逐页发送仅含一张图片的请求并要求每个请求成功完成；全部页成功后按页码顺序无额外换行地合并文本并写入 sidecar。
- **优点**：直接消除单次请求的图像累积；可明确报出失败页；结果只会在全部页成功后写出，能阻止缺页或部分结果；满足跨页连续文本要求。
- **缺点 / 风险**：请求次数与 PDF 页数线性增长，总耗时相应增长。
- **影响面**：`src/reference_utils.py`、`tests/test_prepare_reference.py`、PDF OCR 说明文档和本 issue 记录；不改变 API 请求协议、其他 OCR 后端或上游 JSON 结构。

### 方案 B：按固定页数分批发送

- **做什么**：一次请求发送若干页图片，批次之间合并结果。
- **优点**：请求次数较少。
- **缺点 / 风险**：不同 PDF 页面图像大小差异很大，固定页数无法保证不再超限；批内仍可能缺页，且与用户确认的逐页提交目标不一致。
- **影响面**：与方案 A 相同，但仍保留上下文超限风险。

### 推荐方案

**推荐方案 A**，理由：它以最小改动直接切断上下文累积，并将“渲染页数完整、逐页 OCR 成功、按序合并”固化为成功条件。用户已明确确认逐页提交、完整性校验和跨页连续拼接，因此本分析标记为已确认。
