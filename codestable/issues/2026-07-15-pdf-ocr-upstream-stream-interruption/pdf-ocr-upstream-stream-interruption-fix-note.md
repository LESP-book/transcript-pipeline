# PDF OCR 上游响应流中断修复记录

## 1. 实际采用方案

采用用户确认的方案 C，并以“人工补页重试”代替无依据的固定自动重试次数：

1. 每页 OCR 成功后立即以独立 TXT 检查点原子落盘；空文件也代表已经成功识别的合法空白页。
2. 单页失败只记录该页错误，调度器继续按原有节奏投递和等待其余页面，不再因首个异常提前终止。
3. 一轮结束后汇总全部失败页；只在所有页检查点齐全时，按页码拼接并生成整本 TXT。
4. Web 状态新增总页数、成功页数、失败页码和逐页错误；未补齐的任务显示“待补页”。
5. Web 新增“重试缺失页”入口，在同一任务目录中恢复，只调度没有成功检查点的页面。
6. 应用进程重启后，原本处于 pending/running 的任务标记为可恢复，不伪装为整本成功。

本次没有新增自动重试上限、退避时间或请求超时。已有“每 10 秒投递一页、最多 20 个在途请求”
保持不变；补页时仍沿用同一调度机制。用户可以根据失败页清单重复点击补页，直到全部页面齐全。

## 2. 检查点与兼容规则

- 检查点位于 `data/jobs/pdf-ocr/{task_id}/pages/`，每本书保留原目录层级，并使用
  `{书名}.pages/page-000001.txt` 格式保存页结果。
- 页文件先写 `.pending` 临时文件，再以同目录原子替换为最终文件，避免中断时把半写入文件当成成功页。
- 任务状态 JSON 同样改为原子替换，避免逐页进度更新期间被 Web 轮询读到半个 JSON。
- 新增的状态字段全部是向后兼容字段；旧历史任务仍可显示和重试。
- 旧任务没有页检查点，首次点击重试时只能把全部页面视为缺失页；只有本次修复后启动的任务才能真正
  从已成功页面继续。界面不隐瞒这一数据事实，也不会伪造旧任务的成功页。

## 3. 改动文件清单

- `src/ocr_scheduler.py`：返回成功页与失败页集合；失败后继续调度；保留 10 秒错峰和 20 个在途请求约束。
- `src/reference_utils.py`：加载/写入页检查点、仅渲染缺失页、汇总缺页异常、齐页后生成 TXT。
- `src/pdf_book_ocr.py`：把页级进度和错误提升到书籍结果，建立稳定的书籍检查点目录。
- `src/web/pdf_book_ocr.py`：任务路径增加检查点目录，并兼容恢复旧任务请求参数。
- `src/web/state_store.py`：任务状态改为同目录原子写入。
- `src/web/tasks.py`：持久化实时页进度，区分 success、partial 和 failed。
- `api_server.py`：增加同任务补页接口，并处理服务重启后的可恢复状态。
- `frontend/src/api/client.ts`、`frontend/src/views/PDFBookOCRView.vue`：增加页进度、失败页详情和
  “重试缺失页”按钮。
- `tests/`：覆盖失败后继续、检查点恢复、同任务重试、服务重启恢复和状态字段兼容。
- `README.md`：补充检查点、补页重试和完整 TXT 生成规则。

## 4. 验证结果

- 后端完整回归：`.venv/bin/python -m pytest`，239 项全部通过。
- 前端检查：`npm run build` 通过；TypeScript 校验和 Vite 生产构建成功，仅保留既有的大包体积警告。
- 失败继续测试：第 2 页模拟失败后，第 3 页仍会被调用；一轮结束后同时保留第 1、3 页检查点和
  第 2 页错误，不生成缺页整本 TXT。
- 补页恢复测试：第二轮只渲染缺失的第 2 页；补齐后按 1、2、3 页顺序生成最终文本。
- 真实样例：使用 603 页《中国古代史（上册）》预置 602 个检查点，只缺原故障页 195；实际请求只
  补跑第 195 页，返回 661 字，检查点从 602 增至 603，最终报告 603/603 页完成。
- 浏览器验收：旧失败任务可见“重试缺失页”和成功页保留说明；历史恢复正常，浏览器控制台无错误。
  未点击旧任务按钮，避免因旧任务没有检查点而产生完整 603 页的重复 API 调用。

## 5. 风险与后续观察

- 本修复解决的是应用侧的进度保留和恢复，不会掩盖 codex-lb 的真实传输错误；失败页及原始错误仍
  会完整显示，便于继续定位上游服务。
- 成功页以检查点文件是否存在为准，不按文字长度判断，因此不会误伤合法空白页。
- 当前没有自动无限重试；codex-lb 持续不可用时，本轮会完成其余可执行页面并列出失败页，等待用户
  在服务恢复后再次补页。
