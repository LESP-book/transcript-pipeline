# PDF OCR 上游响应流中断根因分析

## 1. 问题定位

| 关键位置 | 说明 |
|---|---|
| `src/codex_lb_client.py:196-251` | 远程请求由 curl 完整读取 SSE；curl 非零退出或 SSE 中出现 `response.failed` 时直接抛出异常，没有区分可恢复的传输中断与永久业务错误。 |
| `src/codex_lb_client.py:309-370` | `response.failed` 一律转换为 `CodexLBClientError`；本次 `code=upstream_unavailable` 未被标记为可重试。 |
| `src/ocr_scheduler.py:56-80` | 任一页 Future 抛错后记录首个失败、停止投递新页，等待活动请求结束后整体抛错。 |
| `src/reference_utils.py:636-658` | 页结果只在内存中聚合；所有页面成功后才一次性写 sidecar，没有页级检查点或恢复入口。 |
| `src/pdf_book_ocr.py:89-103` | 整本异常被转换成失败条目，输出文件为空。 |

归档检索未发现针对 `upstream_unavailable`、`TransferEncodingError` 或 curl 92 的既有恢复规则；
已有错峰并发设计明确选择“不自动重试、失败不写部分结果”，所以当前行为符合旧设计，但不适合
已经实证会遇到瞬时传输中断的 603 页长任务。

## 2. 失败路径还原

**正常路径**：Web 提交 PDF → 每 10 秒投递一页 → curl 持续接收 SSE → 单页文本进入内存结果槽 →
603 页全部成功 → 按页码拼接并写整本 TXT。

**失败路径**：任务运行到第 195 页 → codex-lb / 其上游的响应流在声明长度尚未传完时中断 →
服务返回 `response.failed(code=upstream_unavailable)` → 页 Future 抛错 → 调度器停止新投递 →
内存中的已完成页不落盘 → 整本任务失败。

**分叉点**：`src/ocr_scheduler.py:60-80` — 所有异常都进入不可恢复的首个失败分支；调用链没有
“仅重试当前页”或“保留成功页、服务恢复后继续”的状态。

## 3. 根因

**根因类型**：外部传输故障为触发原因，本地缺少恢复机制为整本失败的直接放大原因。

**根因描述**：失败事件明确来自 codex-lb/上游传输层，而不是 OCR 页面、Prompt、模型输出或
Responses 文本解析。首次长任务收到 `TransferEncodingError`，随后隔离请求又收到 curl 92 的
HTTP/2 `INTERNAL_ERROR`；codex-lb 重启后，同一第 195 页在 HTTP/1.1 和默认 HTTP/2 下都成功
返回 661 字。因此不能把根因归为第 195 页损坏，也没有证据支持强制 HTTP/1.1。应用侧的问题是
把瞬时 `upstream_unavailable` 与永久失败同等处理，并且没有持久化成功页，导致一次网络流中断
放大为整本重做。

aiohttp 官方文档把“响应体数据不足以满足声明长度”列为读取 payload 时抛出
`ClientPayloadError` 的条件；aiohttp 3.13.4 的官方发布说明也明确指出，当分块编码声明的块长度
与实际收到的数据不匹配时会抛出 `TransferEncodingError`，而不是无限等待永远不会到达的数据。
因此异常文本中的 `400` 是 HTTP payload 解析器对不完整传输的判定，不代表 PDF 请求参数被业务
接口以 HTTP 400 拒绝。外层 Responses 事件已经把它归类为 `server_error / upstream_unavailable`。

失败事件的 `created_at=1784118499` 对应 2026-07-15 20:28:19 +08:00；任务状态在 20:30:06
落为失败。若要继续定位 codex-lb 内部是连接池、代理进程还是具体上游节点异常，需要查询这一时间段
及 response id `ab2cfbc9-521b-4af8-b858-077586592464` 的 codex-lb 服务端日志；当前客户端证据只能
精确定位到“codex-lb 读取/转发上游响应流时数据提前结束”。

**是否有多个根因**：是。

1. **触发原因**：codex-lb 当时处于异常的响应流传输状态；重启后恢复。
2. **主放大因素**：页任务没有针对明确瞬时传输错误的重试策略。
3. **次放大因素**：没有页级检查点，重试耗尽或进程重启后无法从已完成页继续。
4. **观察项**：20 个长连接可能增加代理压力，但隔离的单页请求在重启前同样失败，因此并发不是
   本次故障的必要条件，目前证据不足以把它定为根因。

## 4. 影响面

- **影响范围**：所有通过远程 codex-lb 执行的长 PDF OCR 都可能被一次瞬时流中断终止；页数越多，
  至少遇到一次中断的机会越高。
- **潜在受害模块**：独立 PDF OCR 和阶段 3 的 Codex API PDF OCR共用同一调用链；阶段 6 精修也
  使用流式客户端，但不是页级批任务，恢复策略不能直接照搬。
- **数据完整性风险**：不会写出缺页 TXT，因而没有不完整结果冒充成功；但已完成页只在内存中，
  失败或重启会造成时间与 API 成本损失。
- **严重程度复核**：维持 P1。重启 codex-lb 可绕过当前故障，但数百页核心场景仍缺少可靠恢复能力。

## 5. 修复方案

### 方案 A：保持现状，故障时重启 codex-lb 并整本重跑

- **做什么**：不改代码；把 `upstream_unavailable`、curl 92 作为运维告警，重启服务后重新提交。
- **优点**：零代码风险；本次重启后第 195 页已验证恢复。
- **缺点 / 风险**：以后任一页再次中断都要从头开始，重复此前页面的时间和 API 成本。
- **影响面**：无代码改动，仅增加运维说明。

### 方案 B：仅对明确的瞬时传输错误做页级重试

- **做什么**：给错误增加可重试分类；只对 `upstream_unavailable`、连接中断、curl 18/92 等明确
  传输故障重试同一页，重试次数和退避间隔做成配置并记录日志。内容错误、鉴权错误、模型拒绝等
  仍立即失败。
- **优点**：改动相对小，可自动跨过短暂的代理抖动，不改变最终 TXT 结构。
- **缺点 / 风险**：codex-lb 持续异常时最终仍会整本失败，且此前成功页依然无法恢复；重试配置需要
  依据服务容量确定，不能无界重试。
- **影响面**：`src/codex_lb_client.py` 或 OCR 页 worker、配置模型、调度测试和 README。

### 方案 C：可重试错误 + 页级检查点与恢复（推荐）

- **做什么**：在方案 B 基础上，每页成功后原子写入任务目录；重新提交或恢复任务时只请求缺失页，
  全部页面齐备后再按页码合并整本 TXT。Web 状态显示已完成页、当前活动页和重试页。
- **优点**：同时覆盖短暂网络抖动、codex-lb 重启和应用重启；即使第 195 页最终失败，也无需重做
  前面已成功页面，最符合数百页书籍场景。
- **缺点 / 风险**：改动范围最大，需要定义页结果校验、任务恢复、过期文件清理和状态兼容规则。
- **影响面**：Codex 客户端错误分类、OCR 调度器、PDF OCR 编排、Web 任务状态、测试、配置与文档。

### 推荐方案

**推荐方案 C**。方案 A 已证明能临时恢复服务，但不能保护长任务投入；方案 B 能处理短暂中断，
却仍无法应对持续故障或服务重启。此次任务在第 195 页、运行约 36 分钟后失败，已经为页级检查点
提供了真实需求依据。若希望先做最小修复，也可以先实施方案 B，再单独规划检查点能力。

用户于 2026-07-15 确认采用方案 C，并补充恢复语义：

1. 单页失败后继续投递和执行其余页面，不因首个错误立即结束整本任务。
2. 成功页立即持久化，任务结束时完整列出失败/缺失页及错误详情。
3. Web 增加“重试缺失页”按钮；重试沿用同一任务与检查点，只处理没有成功页文件的页面。
4. 缺失页可以由用户反复重试，不设置无依据的自动重试次数；最终 TXT 仍只在全部页面齐全后生成。
