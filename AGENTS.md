# AGENTS.md

本文件定义本项目中 Codex / coding agent 的长期工作规则。
除非用户明确要求，否则所有实现、修改、测试、评审都必须遵守本文件。

---

## 1. 项目目标

本项目是一个“读书会录屏整理流水线”。

目标是把录屏中的语音内容，逐步处理为适合人工校对的结构化文本，最终形成可用的整理稿。

材料中通常同时包含：

1. 原文朗读
2. 讲解内容
3. 提问 / 讨论 / 问答
4. 开场播报、标题、年月等附加信息

本项目采用分阶段推进策略，不允许一次性把所有阶段混在一起实现。

---

## 2. 总体原则

1. **严格按阶段推进**
   - 只实现当前批准阶段的目标
   - 不要擅自越界实现未来阶段
   - 不要为了“完整”而提前加入 OCR、LLM、Markdown 拼装等未来功能

2. **优先最小修改**
   - 优先选择最小可行修复
   - 不要大规模重构
   - 不要破坏已有中间产物结构

3. **先跑通，再增强**
   - 先保证当前阶段可运行、可测试、可追踪
   - 再做精度和体验增强
   - 不要跳步

4. **中间结果必须保留**
   - 不直接覆盖上游产物
   - 每一阶段都应输出独立中间结果
   - 便于排错、复核和后续加工

5. **规则优先，模型滞后**
   - 在进入 LLM 阶段前，优先把结构化中间层做好
   - 不要让大模型承担本应由工程结构完成的任务

---

## 3. Python 与环境规则

所有 Python 命令必须优先使用项目内 `.venv`。

### 必须使用
- `.venv/bin/python -m pytest`
- `.venv/bin/python scripts/xx.py`

### 不应使用
- 系统 `python`
- 系统 `python3`
- 未激活的外部环境

### 如果 `.venv` 不存在
先提示用户或先执行环境初始化，不要假装已经测试通过。

---

## 4. 每轮任务的固定流程

每一轮阶段任务都必须按以下顺序执行：

1. 阅读当前任务要求
2. 检查当前项目结构与已有阶段状态
3. 只修改当前阶段所需文件
4. 运行 `.venv/bin/python -m pytest`
5. 运行一个真实样例（如果该阶段存在可运行脚本）
6. 检查真实输出是否符合该阶段目标
7. 按 `docs/REVIEW_CHECKLIST.md` 做自我评审
8. 如发现问题，优先自行最小修复
9. 最后再向用户汇报

---

## 5. 汇报格式

每轮结束后，必须按下面格式汇报，不要省略关键项：

### 完成情况
- 本轮实现了什么
- 是否完全符合本阶段目标

### 修改文件
- 列出新增 / 修改的文件

### 关键实现
- 简要说明核心逻辑
- 说明为什么这样设计

### 测试结果
- 是否使用 `.venv/bin/python -m pytest`
- 测试通过情况
- 若失败，失败原因是什么

### 真实运行结果
- 是否执行了真实样例
- 输出目录和核心结果是什么
- 是否存在仍需人工关注的问题

### 范围边界
- 明确说明本轮**没有**实现哪些未来阶段内容

### 下一步建议
- 明确指出下一步最适合做什么
- 不要泛泛而谈

---

## 6. 阶段边界规则

未获批准时，不要实现以下内容：

- OCR
- GPT / Gemini 校对
- final Markdown / docx / 最终稿拼装
- embedding / 向量检索
- 复杂全局动态规划对齐
- 新一轮 ASR 重跑
- 与当前阶段无关的大规模重构

如果用户只要求修某个 bug，则只修该 bug，不要顺手推进下一个阶段。

---

## 7. 输出结构兼容性要求

1. 不要轻易改动已有中间 JSON 的主结构
2. 新增字段时，优先向后兼容
3. 不要回改上游产物
4. 每一阶段输出必须继续可被下游消费
5. 如果必须调整字段名，必须明确说明原因和影响

---

## 8. 评审要求

完成任何阶段任务后，必须阅读并遵守：

- `docs/REVIEW_CHECKLIST.md`

不得只报告“代码已完成”，必须报告：

- 是否真实测试
- 是否真实运行
- 是否存在已知偏差
- 是否建议继续推进

---

## 9. 当前项目的阶段推进方式

以 `docs/PIPELINE_PLAN.md` 为准。

如果用户说“进入下一阶段”，应先查看该文件中当前阶段状态，再执行对应任务。
如果发现当前阶段结果仍明显不适合作为下一阶段输入，应优先建议做小范围修正，而不是盲目前进。

---

## 10. 优先级排序

发生取舍时，优先级如下：

1. 正确性
2. 范围控制
3. 可运行
4. 可测试
5. 可追踪
6. 可解释
7. 美观 / 花哨

---

## 11. 对真实样例的态度

测试通过不等于阶段质量已经足够。

如果该阶段会生成中间 JSON / 文本结果，必须尽量检查一个真实样例。
如果真实样例明显说明逻辑偏差，不能只靠“pytest 通过”宣布阶段成功。

---

## 12. 禁止事项

1. 不要伪造已通过的测试
2. 不要伪造已跑通的真实样例
3. 不要把“计划中的功能”说成“已经实现”
4. 不要为了省事跳过 `.venv`
5. 不要把明显有问题的结果硬说成“可以进入下一阶段”
6. 不要把用户未批准的未来阶段提前做掉

---

## 13. 默认任务完成定义

只有同时满足以下条件，才能说“本轮完成”：

1. 当前阶段目标已实现
2. `.venv/bin/python -m pytest` 通过
3. 真实样例运行成功（如适用）
4. 输出结果基本符合当前阶段预期
5. 无明显越界实现
6. 已给出下一步建议
